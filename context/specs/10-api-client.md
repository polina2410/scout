# Spec 10 — API Client

**Plan ref:** Phase 6, Step 10  
**Goal:** A typed, tested fetch layer that attaches the API key on every request, parses response bodies into generated schema types, and surfaces a discriminated `AsyncState<T>` union for components to render loading / error / empty states.

---

## 1. What is already in place — do not redo

- `src/api/generated/schema.ts` — auto-generated by `pnpm generate`; do not modify by hand
- Vite dev proxy: `/api/*` → `http://localhost:8080/*` (strips `/api` prefix)
- `VITE_API_URL` and `VITE_API_KEY` in `.env.local`; consumed via `import.meta.env`

---

## 2. Convenience type aliases — `src/api/types.ts`

Extract named aliases from the generated schema so consumers never reference the verbose `components["schemas"][...]` path. These are the only types consumers should import from this layer.

```typescript
import type { components, operations } from './generated/schema'

export type Photo = components['schemas']['Photo']
export type Prediction = components['schemas']['Prediction']
export type BoundingBox = components['schemas']['BoundingBox']
export type PhotoPage = components['schemas']['PhotoPage']

export type ListPhotosParams = NonNullable<
  operations['listPhotos']['parameters']['query']
>
```

No other types are needed in this file. Do not re-export the full `components` or `operations` objects.

## 3. `ApiError` class — `src/api/client.ts`

A typed error carrying the HTTP status, machine-readable code, human message, and correlation id from the backend error body:

```typescript
export class ApiError extends Error {
  constructor(
    public readonly status: number,
    public readonly code: string,
    message: string,
    public readonly requestId: string,
  ) {
    super(message)
    this.name = 'ApiError'
  }
}
```

---

## 4. Core `apiFetch` wrapper — `src/api/client.ts`

```typescript
async function apiFetch<T>(path: string, init?: RequestInit): Promise<T>
```

Behaviour:

- Prepends `import.meta.env.VITE_API_URL` to `path`; path must start with `/`
- Merges `X-API-Key: import.meta.env.VITE_API_KEY` into every request's headers
- On non-2xx: parses the JSON error body and throws `ApiError(status, body.code, body.message, body.request_id)`; if the body cannot be parsed, throws `ApiError(status, 'UnknownError', 'Unexpected response', '')`
- On network failure (fetch rejects): re-throws as-is; callers handle `Error` separately from `ApiError`
- Always sets `Content-Type: application/json` on requests that carry a body (i.e., when `init.body` is present)

`apiFetch` is not exported — only the named endpoint functions are part of the public surface.

## 5. `listPhotos` — `src/api/client.ts`

```typescript
export async function listPhotos(params?: ListPhotosParams): Promise<PhotoPage>
```

- Calls `GET /photos` via `apiFetch`
- Serialises `params` into query string; omits keys whose value is `undefined`; if `minConfidence` is `0`, omit it (the backend default is 0 — sending it is harmless but omitting keeps URLs clean)
- Returns the full `PhotoPage` (with `items` array and optional `next_token`)

---

## 6. `getPhoto` — `src/api/client.ts`

```typescript
export async function getPhoto(photoId: string): Promise<Photo>
```

- Calls `GET /photos/{photoId}` via `apiFetch`
- `photoId` is interpolated directly into the path; it is a UUID from the database and contains no characters that need escaping

---

## 7. `AsyncState<T>` — `src/api/types.ts`

A discriminated union that every data-fetching hook returns. Components switch on `status` to render the correct state without a blank screen.

```typescript
export type AsyncState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; message: string }
```

Export `AsyncState` from `src/api/types.ts`. No initial state helper is needed — callers write `{ status: 'idle' }` directly.

---

## 8. Barrel export — `src/api/index.ts`

Create a barrel that re-exports everything consumers need:

```typescript
export type { Photo, Prediction, BoundingBox, PhotoPage, ListPhotosParams, AsyncState } from './types'
export { listPhotos, getPhoto, ApiError } from './client'
```

The generated `schema.ts` is never imported by components directly.

## 9. Tests — `src/api/client.test.ts`

Mock `globalThis.fetch` using `vi.stubGlobal('fetch', vi.fn())` before each test; restore with `vi.unstubAllGlobals()` after.

| Test | Arrange | Assert |
|---|---|---|
| `listPhotos returns PhotoPage on 200` | fetch resolves `{ ok: true, status: 200, json: () => photoPage }` | resolves to the `photoPage` object |
| `listPhotos serialises params into query string` | call with `{ classId: 'mirid', minConfidence: 0.8, cursor: 'tok' }` | the URL passed to fetch contains `classId=mirid&minConfidence=0.8&cursor=tok` |
| `listPhotos omits minConfidence=0` | call with `{ minConfidence: 0 }` | URL does NOT contain `minConfidence` |
| `listPhotos throws ApiError on 401` | fetch resolves `{ ok: false, status: 401, json: () => authError }` | rejects with `ApiError` where `status === 401` and `code === 'AuthenticationRequired'` |
| `listPhotos throws ApiError on 500` | fetch resolves `{ ok: false, status: 500, json: () => serverError }` | rejects with `ApiError` where `status === 500` |
| `getPhoto returns Photo on 200` | fetch resolves with photo body | resolves to photo |
| `getPhoto throws ApiError on 404` | fetch resolves `{ ok: false, status: 404, json: () => notFoundError }` | rejects with `ApiError` where `status === 404` |
| `apiFetch attaches X-API-Key header` | call `listPhotos()` | fetch was called with a headers object containing `X-API-Key` |

Each test sets up `import.meta.env.VITE_API_URL` and `VITE_API_KEY` via Vitest's env injection or by directly stubbing `import.meta.env` before the import under test.
